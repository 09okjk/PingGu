## 一、背景与目标

### 背景

服务贸易人员根据客户的原始需求制作**服务需求单**，评估人员需要根据需求单内容人工完成**服务评估报告**。现有流程依赖评估人员的个人经验，效率较低，且经验难以沉淀传承。

### 目标

构建一个**智能评估 Agent**，作为上层 Agent 的编排模块（参考 OpenClaw Skill 模式），通过多个细粒度 Skill 协作，实现：

- 自动读取并理解服务需求单（含非结构化备注/原始邮件）
- 渐进式检索历史相似评估案例，辅助生成评估报告草稿
- 对每个生成字段标注置信度，引导评估人员高效审核
- 随使用积累持续提升推荐精准度（数据飞轮）

### 当前数据基础

| 项目 | 情况 |
|------|------|
| 历史评估数据 | 8000 条，xlsx 格式，文件大小约 2.46MB |
| 数据特殊结构 | 部分列存储 JSON 格式内容 |
| 备注语言 | 英文为主，含简体中文、繁体中文、韩语、日语 |
| 员工能力数据 | 暂无，评估人员凭经验判断，二期接入 |

## 二、业务流程概述

```
客户原始需求（邮件等）
        ↓
服务贸易人员制作【服务需求单】
        ↓
评估人员提交需求单至 Agent
        ↓
智能评估 Agent 编排处理
├── ParseRequirementSkill：解析结构化字段 + NLU 备注
├── SearchHistoryCasesSkill：渐进式检索历史相似案例
├── MatchRisksSkill：多维风险匹配
├── EstimateWorkHoursSkill：工时估算
├── EstimateManpowerSkill：人力调度推理
└── GenerateReportSkill：整合结果，生成评估报告草稿（含置信度标注）
        ↓
评估人员审核 & 修改
        ↓
最终评估报告确认输出
        ↓
差异回流 → 历史数据库更新（飞轮）
```

## 三、服务需求单字段定义（输入）

| # | 字段名 | 类型 | 说明 |
|---|--------|------|------|
| 1 | 服务描述 | 枚举（有穷） | 简单描述需要服务的内容 |
| 2 | 服务类型 | 枚举（有穷） | 如：维修、更换、检测、安装等 |
| 3 | 业务归口 | 枚举（有穷） | 负责此次服务的业务部门 |
| 4 | 所属设备名称 | 枚举（有穷） | 服务设备所属的一级设备名称 |
| 5 | 所属设备型号 | 级联枚举（依据字段4） | 对应一级设备下的具体型号 |
| 6 | 所属设备厂家 | 文本 | 一级设备的生产厂家 |
| 7 | 服务设备单位 | 枚举 | 计量单位，如：个、台、把等 |
| 8 | 服务设备数量 | 数字 | 本次需要服务的具体目标设备数量 |
| 9 | 服务设备型号 | 文本（多条） | 与数量对应，每个设备的型号 |
| 10 | 备注 | 长文本（多语言） | 补充内容 + 客户原始邮件，辅助评估判断 |

## 四、服务评估报告字段定义（输出）

### 4.1 风险提示

- 从已有风险库中选择匹配项
- 若风险库中无合适条目，评估人员可手动输入新增
- 支持多条

### 4.2 施工任务

- 支持多条
- 每条内容：具体的工作内容描述，用于指导施工人员

### 4.3 施工人数及工时

支持多条，每条包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| 详细工作内容 | 文本 | 具体工作细项描述 |
| 单位 | 枚举（选填） | 个、台、把等 |
| 数量 | 数字 | 面向的设备数量 |
| 工种 | 枚举（有穷） | 如：电气工程师、轮机工程师等 |
| 职级 | 枚举（按工种细分，有穷） | 不同工种有不同职级体系 |
| 人数 | 数字 | 该项工作所需人数 |
| 单人工时 | 数字（小时） | **注意**：指整体施工从开始到结束的时长，非个人完成该任务的时间。即使该任务提前完成，只要整体施工未结束，工时持续计算。 |

### 4.4 总计

| 字段 | 类型 | 说明 |
|------|------|------|
| 是否航修 | 布尔 | 是否属于航修类型 |
| 总小时数 | 数字 | 施工开始到结束的总时长，**非所有人工时相加** |
| 总人数 | 数字 | 最少所需总人数，**非各任务人数直接相加**（见第八节专项说明） |

### 4.5 需要工具

支持多条，每条包含：

| 字段 | 类型 |
|------|------|
| 工具名称 | 文本 |
| 工具类型 | 枚举（有穷） |
| 计量单位 | 枚举 |
| 需要数量 | 数字 |

### 4.6 耗材

支持多条，每条包含：

| 字段 | 类型 |
|------|------|
| 耗材名称 | 文本 |
| 耗材型号 | 文本（必填） |
| 计量单位 | 枚举 |
| 数量 | 数字 |

### 4.7 专用工具

支持多条，每条包含：

| 字段 | 类型 |
|------|------|
| 专用工具名称 | 文本 |
| 专用工具型号 | 文本 |
| 计量单位 | 枚举 |
| 数量 | 数字 |

### 4.8 设备/备件需求

- 描述本次服务需要提供的设备或备件
- 注意：若需求方自身有多余设备或备件，则无需填写（我方不提供）

## 五、整体架构设计

### 架构定位

这是一个**知识增强型结构化推理 Agent**，由上层编排层调度多个细粒度 Skill 协作完成评估任务。核心能力层次如下：

```
┌─────────────────────────────────────────────────────────────┐
│                     智能评估 Agent（编排层）                  │
│                                                             │
│   ┌─────────────┐    ┌──────────────┐    ┌──────────────┐   │
│   │  理解与提取  │ →  │  检索与匹配  │ →  │   推理与生成  │   │
│   │  (NLU 层)   │    │ (Agentic     │    │ (Generation) │   │
│   │             │    │  Search 层)  │    │              │   │
│   └─────────────┘    └──────────────┘    └──────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
         ↑ 输入：服务需求单                ↓ 输出：评估报告草稿
```

### Agent 在体系中的位置

```
上层 Agent（对话/工作流编排）
        ↓ 调用
┌──────────────────────────────────────────────┐
│            智能评估 Agent（编排层）             │
│                                              │
│  ┌──────────────────────────────────────┐   │
│  │             Skills（主动执行）         │   │
│  │  ParseRequirementSkill               │   │
│  │  SearchHistoryCasesSkill（Agentic）   │   │
│  │  MatchRisksSkill                     │   │
│  │  EstimateWorkHoursSkill              │   │
│  │  EstimateManpowerSkill               │   │
│  │  GenerateReportSkill                 │   │
│  └──────────────────────────────────────┘   │
│                                              │
│  ┌──────────────────────────────────────┐   │
│  │         References（静态注入）         │   │
│  │  R2 枚举字典与业务规则库               │   │
│  │  R4 工具/耗材推荐模板（二期可升级）     │   │
│  └──────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
        ↓ 返回
结构化评估报告（含置信度标注）
```

### Skills vs References 划分原则

| 类型 | 适用场景 | 特征 |
|------|---------|------|
| **Skill** | 需要多步查询、计算逻辑、动态交互 | 主动执行，有内部逻辑 |
| **Reference** | 静态查表、枚举校验、模板套用 | 被动注入 context，内容固定 |

### S1：SearchHistoryCasesSkill ⭐ 最高优先级

**定位**：历史相似案例的渐进式 Agentic 检索，Agent 推理的最核心依据。  
**原形式**：R1 Reference → **升级为 Skill**。  
**数据来源**：现有 8000 条历史评估 xlsx 数据。  
**实施优先级**：MVP 必须。

#### 为何从 RAG Reference 升级为 Agentic Search Skill

| 对比维度 | RAG Reference | Agentic Search Skill |
|---------|--------------|---------------------|
| 检索方式 | 单次检索，Top-K 塞入 context | 多步渐进，根据中间结果调整策略 |
| 候选集控制 | 无法动态调整 | 先宽筛后精排，自动扩大/收窄条件 |
| Context 占用 | 8000 条输出侧内容复杂，Top-K 极易过长 | 按需返回，只传必要字段 |
| 适合数据类型 | 非结构化文本块 | 结构化记录（本场景更适合） |

#### 后端存储层设计

```
历史评估数据库（S1 背后的存储层）
│
├── 关系型数据库（精确过滤 + 分页）
│   推荐：PostgreSQL
│   用途：按服务类型、设备名称、设备型号精确过滤
│         支持分页、排序、聚合统计
│
├── 向量数据库（语义相似度检索）
│   推荐：Qdrant / Weaviate / pgvector（若已用 PG 可合并）
│   用途：备注/施工任务描述的多语言 embedding 检索
│         跨语言匹配（日语输入 → 英文历史记录）
│
└── 全文检索引擎（关键词渐进查找，可选，作为补充召回）
    推荐：PostgreSQL FTS 或 Elasticsearch
    用途：特殊型号、厂家名等关键词的精确命中
```

#### 渐进式 Agentic 检索流程

```
Step 1：结构化粗筛（关系型 DB）
  → 按「服务类型 × 设备名称」过滤，得候选集（几十~几百条）
  → 若候选集 < 阈值，自动放宽到仅按「服务类型」重筛

Step 2：语义精排（向量 DB）
  → 对候选集的备注/任务描述做多语言向量相似度排序
  → 选 Top-K 条（K 默认 5，可由上层 Agent 配置）

Step 3：关键词验证（全文检索，可选）
  → 对特殊型号、厂家名做精确命中确认
  → 提升结果可解释性（可标注"命中关键词：XXX"）

Step 4：结果整合 + 摘要压缩
  → 对 Top-K 条记录的输出侧内容做摘要，避免 context 过长
  → 返回：案例 ID、相似度分数、关键字段摘要、来源标注
```

#### 数据结构设计

```
历史评估记录
├── 输入侧索引（用于检索）
│   ├── 服务描述
│   ├── 服务类型
│   ├── 设备名称 + 型号
│   └── 备注摘要（语义向量化，支持多语言）
└── 输出侧内容（用于参考生成）
    ├── 风险提示列表
    ├── 施工任务列表
    ├── 施工人数及工时明细
    ├── 工具 / 耗材 / 专用工具清单
    └── 最终总计（总工时、总人数）
```

#### 数据入库流程

```
原始 xlsx 数据
    ↓
Step 1：列结构分析
  - 梳理所有列名及数据类型
  - 识别 JSON 列并解析其内部结构
  - 统计各字段的填写率和枚举分布
    ↓
Step 2：数据清洗
  - JSON 列展开，保留原始嵌套层级
  - 枚举值归一化（修正历史不规范录入）
  - 多语言内容标记 language 字段
  - 空值处理（标记而非删除）
    ↓
Step 3：知识挖掘（自动生成其他库的种子数据）
  - 提取高频风险组合 → S2 风险库种子数据
  - 统计工具/耗材频次 → R4 推荐模板种子数据
  - 分析工时分布 → S3 工时经验数据
    ↓
Step 4：多路索引建立（并行）
  ├── 结构化索引写入 PostgreSQL
  │   └── 服务类型 × 设备名称 × 设备型号 组合键
  └── 向量索引写入向量 DB
      ├── 备注/邮件内容向量化（多语言 embedding）
      └── 施工任务描述向量化
```

> ⚠️ **待确认**：xlsx 中的具体列名及 JSON 列的内部字段结构，将直接影响数据处理方案和索引设计。

### S2：MatchRisksSkill ⭐ 最高优先级

**定位**：多维度风险条目匹配推荐。  
**原形式**：R3 Reference → **升级为 Skill**。  
**升级原因**：风险匹配需要服务类型 + 设备类型 + 备注关键词三路并行匹配，并对结果排序、去重、置信度计算，逻辑复杂，超出静态 Reference 的适用范围。  
**数据来源**：从历史评估数据中提取 + 业务专家人工补充。  
**实施优先级**：MVP 必须。

#### 风险条目结构

```
风险条目
├── 风险 ID
├── 风险名称（展示给用户）
├── 风险描述（详细说明）
├── 触发条件
│   ├── 关联服务类型列表
│   ├── 关联设备类型列表
│   └── 关键词触发列表（从备注中识别）
├── 风险等级（高 / 中 / 低）
└── 建议措施
```

#### Skill 执行逻辑

```
输入：服务类型、设备类型、备注文本

Step 1：结构化匹配
  → 按触发条件中的服务类型 + 设备类型过滤候选风险条目

Step 2：关键词扫描
  → 对备注文本做关键词命中检测（触发条件中的关键词列表）

Step 3：排序与置信度计算
  → 三路命中（类型 + 设备 + 关键词）→ 高置信
  → 双路命中 → 中置信
  → 单路命中 → 低置信，提示人工确认

输出：匹配风险条目列表（含置信度、触发依据说明）
```

**飞轮回流**：评估人员手动新增的风险条目自动回流至风险库，持续丰富覆盖范围。

### S3：EstimateWorkHoursSkill ⭐ 中优先级

**定位**：基于历史统计数据的工时区间估算。  
**原形式**：R5 Reference → **升级为 Skill**。  
**升级原因**：工时估算涉及修正因子计算（设备数量系数、首次/重复施工、P50/P80/P95 选择），是计算逻辑而非单纯查表，适合封装为 Skill。  
**数据来源**：从历史评估数据中统计分析。  
**实施优先级**：第二期。

#### 工时经验数据结构

```
工时经验条目
├── 适用服务类型
├── 适用设备类型
├── 工作内容标签
├── 工种
├── 历史工时分布
│   ├── P50（中位数，常规情况）
│   ├── P80（较复杂情况）
│   └── P95（复杂情况）
└── 工时修正因子
    ├── 设备数量系数
    ├── 设备老旧程度
    └── 首次施工 vs 重复施工
```

#### Skill 执行逻辑

```
输入：工作内容标签、工种、设备类型、设备��量、施工历史信息

Step 1：查询匹配的工时经验条目（关系型 DB）

Step 2：应用修正因子计算工时区间
  → 基准工时 × 设备数量系数 × 首次/重复系数

Step 3：输出工时区间建议
  → 给出 P50 / P80 / P95 三档，并说明推荐使用哪档的理由
```

### S5：ParseRequirementSkill ⭐ 最高优先级

**定位**：需求单结构化字段解析 + 备注 NLU（纯推理 Skill）。  
**实施优先级**：MVP 必须。

#### 执行逻辑

```
输入：原始需求单

Step 1：结构化字段解析
  → 枚举合法性校验（对照 R2 枚举字典）
  → 标记非法或缺失字段

Step 2：备注 NLU
  → 提取关键设备信息（型号、故障描述）
  → 识别特殊要求或约束条件
  → 检测时间紧迫性信号
  → 语言识别（决定后续输出语言）

输出：结构化解析结果 + NLU 提取摘要
```

### S6：GenerateReportSkill ⭐ 最高优先级

**定位**：整合所有 Skill 输出结果，合成最终评估报告草稿（编排/生成 Skill）。  
**实施优先级**：MVP 必须。

#### 执行逻辑

```
输入：S1~S5 各 Skill 的输出结果 + R2 枚举约束

Step 1：结果整合与冲突检测
  → 检查各 Skill 结果之间是否存在矛盾（如工时与工种不匹配）

Step 2：逐块生成报告
  ① 风险提示（基于 S2 结果，按置信度排序）
  ② 施工任务拆解（基于 S1 历史案例参考）
  ③ 各任务工种/职级/人数/工时（基于 S1 + S3）
  ④ 总计（调用 S4 求解最优总人数）
  ⑤ 工具/耗材/专用工具推荐清单（基于 R4 模板）

  → 🟢 高置信（直接引用历史相似案例）→ 推荐采纳
  → 🟡 中置信（推断得出，有依据）→ 提示确认
  → 🔴 低置信（无参考依据）→ 提示人工填写

输出：完整评估报告草稿（结构化 JSON，含置信度和来源标注）
```

### R2：枚举字典与业务规则库 ⭐ 最高优先级（保持 Reference）

**定位**：所有有穷枚举的权威数据源 + 业务约束规则定义。  
**形式**：**保持 Reference（静态注入 context）**。  
**保持原因**：枚举数据量小、内容静态、只需查表校验，无需动态执行逻辑，直接注入 context 更轻量高效。  
**数据来源**：业务系统导出 / 人工整理。  
**实施优先级**：MVP 必须，应最先完成。

#### 枚举内容清单

| 枚举类别 | 说明 |
|---------|------|
| 服务描述枚举 | 所有合法的服务描述选项 |
| 服务类型枚举 | 维修、更换、检测、安装等 |
| 业务归口部门 | 各部门名称及职责范围 |
| 设备名称 → 型号映射 | 级联关系映射表 |
| 工种枚举 | 电气工程师、轮机工程师等 |
| 工种 → 职级映射 | 每种工种下的职级体系（各工种独立） |
| 工具类型枚举 | 工具的分类体系 |
| 计量单位枚举 | 个、台、把、套等 |

#### 业务规则示例

```

- 高职级人员可以承担低职级人员的所有任务
- 跨工种不可替代（电气工程师 ≠ 轮机工程师）
```

## 八、总人数计算专项设计

> 这是 Agent 中逻辑最复杂的部分，本质是一个**带职级覆盖约束的人力调度优化问题**，由 `EstimateManpowerSkill` 负责执行。

### 核心规则

- 总人数 ≠ 各任务人数直接相加
- 高职级人员完成串行任务后，可接续承担低职级任务，从而节省人力
- 跨工种不可替代

### 计算模型

```
输入：
  - 任务列表：
    [{工作内容, 工种, 最低职级要求, 所需人数, 持续工时, 时间窗口}]
  - 职级能力覆盖表（来自 R2）
  - 总工时上限

推理步骤：
  1. 识别任务间时间重叠关系 → 重叠任务必须独立人员
  2. 识别串行可行的任务组合 → 可共享人员
  3. 在职级覆盖规则约束下，求最小人员配置

输出：
  - 最小总人数方案
  - 推理过程说明（供评估人员审核）
  - 备选方案（人数+1 的保守方案，附风险说明）
```

### 示例说明

```
场景：
  - 任务1：高职级电气工程师，2人，持续2小时，仅高职级可做
  - 任务2：电气工程师（高/低职级均可），持续6小时，需3人
  - 总工时：10小时（任务1和任务2串行）

推理过程：
  - 任务1结束后，2名高职级工程师���闲，可接续承担任务2
  - 任务2原需3人，高职级接续承担后，仅需补充1名低职级
  - 最小总人数 = 2（高职级）+ 1（低职级）= 3人

对比直接相加：任务1需2人 + 任务2需3人 = 5人（高估2人）
```

### 当前阶段降级策略

由于员工个人能力数据暂不可用（S7 未建设），当前阶段：
- 仅基于职级规则进行**理论最优解**计算
- 给出推理过程，由评估人员结合对员工的了解手动调整
- S7 接入后，此模块自动升级，无需重构

## 九、多语言处理策略

**支持语言**：英文（主）、简体中文、繁体中文、韩语、日语

### 核心方案

使用**多语言 Embedding 模型**（推荐：`multilingual-e5-large` 或 OpenAI `text-embedding-3-large`），一个模型覆盖全部语言，无需分语言维护多套索引。

### 处理流程

```
输入层：保留原文，不做翻译
    ↓
Embedding 层：多语言模型统一向量化
    ↓
检索层：跨语言语义匹配
（例：日语输入 → 可匹配英文历史记录中的相似案例）
    ↓
生成层：输出语言跟随输入语言
（用户用什么语言提交需求单，AI 用什么语言回应）
```

### 业务价值

不同语言的客户需求单，可以无缝匹配到任意语言的历史案例，最大化历史数据的利用价值。

## 十、历史评估数据库处理策略

> 本节描述 S1 SearchHistoryCasesSkill 后端数据的入库处理流程，已整合至第六节 S1 的「数据入库流程」中。

### 数据现状

| 项目 | 情况 |
|------|------|
| 数据量 | 8000 条 |
| 格式 | xlsx |
| 文件大小 | 约 2.46MB |
| 特殊结构 | 部分列存储 JSON 格式内容 |

### 处理步骤概要

```
Step 1��列结构分析
  - 梳理所有列名及数据类型
  - 识别 JSON 列并解析其内部结构
  - 统计各字段的填写率和枚举分布

Step 2：数据清洗
  - JSON 列展开，保留原始嵌套层级
  - 枚举值归一化（修正历史不规范录入，如大小写、别名等）
  - 多语言内容标记 language 字段
  - 空值处理策略（标记而非删除，保留不完整记录的部分价值）

Step 3：知识挖掘（自动从历史数据生成各库种子数据）
  - 提取高频风险组合 → S2 风险库种子数据
  - 统计服务场景对应的工具/耗材频次 → R4 推荐模板种子数据
  - 分析工时分布 → S3 工时经验数据

Step 4：多路索引建立（并行）
  - 结构化索引（PostgreSQL）：服务类型 × 设备名称 × 设备型号 组合键
  - 向量索引（向量 DB）：备注/施工任务描述的多语言向量
  - 全文索引（可选）：关键词补充召回
```

> ⚠️ **待确认**：xlsx 中的具体列名及 JSON 列的内部字段结构，将直接影响 Step 1~2 的实施细节。

## 十一、Skill 接口设计

### 对外暴露的接口

| 接口 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `assess(需求单)` | 完整需求单对象 | 完整评估报告草稿（含置信度） | 主接口，完整编排流程 |
| `search_similar(需求单)` | 需求单对象 | Top-K 历史相似案例列表 | 单独调用 S1，仅检索不生成 |
| `match_risks(服务类型, 设备, 备注)` | 三个字段 | 匹配的风险条目列表（含置信度） | 单独调用 S2 |
| `estimate_manpower(任务列表)` | 任务列表 | 最优人数方案 + 推理说明 | 单独调用 S4 |
| `estimate_workhours(任务标签, 工种, 设备信息)` | 任务信息 | 工时区间建议（P50/P80/P95） | 单独调用 S3（第二期） |

### 输出格式设计原则

- 每个字段附带 `confidence` 置信度（`high` / `medium` / `low`）
- 每个字段附带 `source`（来源历史记录 ID 或推理依据说明）
- 结构化 JSON 输出，便于上层 Agent 进一步处理或前端渲染

## 一、Skill 定位与职责

| 项目 | 内容 |
|------|------|
| **Skill 名称** | ParseRequirementSkill |
| **核心职责** | 将用户原始自然语言邮件解析、拆分、修订并确认成一个或多个标准化需求单 |
| **输入形式** | 自然语言邮件 / 文本描述 |
| **输出形式** | 标准化 RequirementItem 列表 + 状态 + 追问项 + 修订历史 |
| **调用方** | 上层 Agent 主流程 |
| **实施阶段** | MVP 必须 |

### 职责升级

S5 不再只是“解析一次”，而是一个完整交互闭环：

1. 初始解析
2. 用户确认/修改
3. 系统吸收反馈并修订
4. 最终确认
5. 输出给后续 Skill

---

## 二、为什么必须升级为交互闭环版

### 根本原因

当前输入不是标准表单，而是自然语言邮件，并且：

- 可能包含多个服务项
- 术语表达不标准
- 可能存在中英文、缩写、口语化描述
- 枚举体系当前还不完整
- 用户最清楚真正想表达的需求

因此，S5 的最佳形式不是“一次解析完毕”，而是：

> “先给出初版需求单，再通过交互逐步修正到可进入后续流程的稳定版本”

### 升级后的价值

- 前置消除歧义，避免 S1/S2 基于错误输入继续推理
- 用户修改天然形成高质量监督信号
- 为后续 alias / 规则 / few-shot 样例优化提供飞轮数据

## 三、输入现实与设计前提

### 已确认前提

1. 输入主要是自然语言邮件，不是标准表单
2. 文本长度一般不长，通常几百个单词以内
3. 当前阶段不处理附件，但要保留扩展性
4. 一封邮件可能包含多个服务项
5. 每个服务项最终会对应一个需求单
6. LLM 不单独由 S5 配置，而应复用上层 Agent 当前模型能力

---

## 四、整体流程定位

### 在总流程中的位置

```text
原始客户邮件
    ↓
S5 ParseRequirementSkill
    ↓
输出：已确认或待确认的 RequirementItem[]
    ↓
确认完成后，对每个 RequirementItem 并行调用：
  - S1 SearchHistoryCasesSkill
  - S2 MatchRisksSkill
  - 其他后续 Skill
    ↓
S6 GenerateReportSkill
```

### 关键原则

**S5 必须先于 S1 执行。**

因为 S1 的检索输入依赖 S5 输出的结构化字段。

## 五、三阶段能力设计

### 5.1 Parse 阶段：初始解析

输入：
- 原始邮件文本

输出：
- `session_id`
- `requirements`
- `status`
- `next_questions`
- `revision_history = []`

职责：
1. 识别有几个服务项
2. 拆分需求
3. 提取核心字段
4. 映射 R2 枚举
5. 标记歧义和待确认项

---

### 5.2 Revise 阶段：修订更新

输入：
- `session_id`
- 当前 requirement 草稿
- 用户反馈文本
- revision_history

输出：
- 更新后的 requirements
- 新的 next_questions
- 增量 revision_history
- 新状态

职责：
1. 吸收用户修正意见
2. 更新字段值
3. 判断是否仍需继续询问

---

### 5.3 Confirm 阶段：最终确认

输入：
- `session_id`
- 当前 requirement 列表
- 用户确认语句（可选）

输出：
- `status = confirmed`
- 最终 requirements
- 完整 revision_history

职责：
1. 结束交互
2. 固化最终需求单
3. 准备进入后续 S1/S2/S6

---

## 六、输出结构设计

### 顶层结构

```json
{
  "session_id": "sess-xxx",
  "status": "needs_confirmation",
  "action": "parse",
  "requirements": [],
  "next_questions": [],
  "revision_history": [],
  "metadata": {}
}
```

### 状态定义

| 状态 | 含义 |
|------|------|
| `draft` | 已完成初步解析，基本无明显歧义 |
| `needs_confirmation` | 仍需要用户确认或修订 |
| `confirmed` | 用户已确认最终版本 |

### RequirementItem 核心字段

- `requirement_id`
- `summary`
- `business_type`
- `service_desc`
- `service_type`
- `equipment_name`
- `equipment_model`
- `equipment_quantity`
- `equipment_unit`
- `remark`
- `original_evidence`
- `ambiguities`
- `confidence`
- `needs_user_confirmation`

## 七、交互确认机制设计

### 7.1 交互原则

不是反复问“对不对”，而是结构化确认：

1. 先确认拆分是否正确
2. 再确认关键检索字段
3. 最后确认补充字段
4. 最终确认是否进入后续流程

### 7.2 问题类型

#### 类型 A：拆分确认
- 我识别出 2 个服务项，是否正确？
- 是否需要新增、删除、合并或拆分？

#### 类型 B：关键字段确认
- 第 2 项服务类型识别为“维修”，是否需要改为“更换”？
- 第 1 项设备是否为“主机”？

#### 类型 C：缺失字段补问
- 第 1 项是否有设备型号？
- 数量是否为 1 台？

#### 类型 D：结束确认
- 当前需求单是否确认完成，可进入后续评估流程？

## 八、修订与确认逻辑

### 8.1 revise 的本质

revise 不是重新从头 parse，而是：

- 以当前 requirement 草稿为基础
- 吸收用户反馈
- 更新特定字段或状态

### 8.2 confirm 的停止条件

满足以下任一即可：

1. 用户明确表示“确认 / 无需修改 / 可以了”
2. 所有关键字段已齐备，且用户接受当前版本
3. 上层 Agent 决定带低置信继续

## 九、与 R2 / S1 / S2 / S6 的关系

### 与 R2
- R2 是全局共享 Reference
- S5 按需读取其子集：
  - 服务描述枚举
  - 服务类型枚举
  - 业务归口枚举
  - 设备名称枚举
  - 单位枚举
  - alias 词典

### 与 S1
S5 输出的以下字段直接喂给 S1：
- `business_type.code`
- `service_desc.code`
- `service_type.code`
- `equipment_model.code/name`
- `remark`

### 与 S2
S5 输出的以下字段可直接用于风险匹配：
- `service_desc`
- `service_type`
- `remark`

### 与 S6
S6 汇总每个已确认 RequirementItem 的后续结果，生成多服务项评估结果。

## 十、开发阶段设计

### 10.1 当前开发阶段目标

目标不是一步到位的“强智能自动学习”，而是：

1. 把 parse / revise / confirm 跑通
2. 输出结构稳定
3. 支持本地测试
4. 支持多服务项拆分
5. 支持修订历史记录
6. 支持 next_questions 生成

### 10.2 当前实现策略

- 规则优先
- 枚举 / alias 匹配优先
- 不强依赖 LLM
- 输入输出 JSON 规范化
- 本地文件形式保存 reference

### 10.3 当前不做的内容

- 自动持久化数据库
- 自动写回正式规则库
- 自动附件解析
- 自动学习更新 alias
- 复杂多轮语义修订

## 十一、生产阶段设计

### 11.1 生产环境流程

```text
用户邮件输入
    ↓
S5 parse
    ↓
输出初版需求单 + next_questions
    ↓
用户修订
    ↓
S5 revise
    ↓
若仍需确认，继续交互
    ↓
用户确认
    ↓
S5 confirm
    ↓
进入 S1 / S2 / S6
```

### 11.2 生产初期建议

建议采用“**先确认，再进入后续 Skill**”模式，而不是解析完立即进入 S1。

原因：
- 生产初期最大风险是需求拆分和字段映射错误
- 提前确认成本最低

### 11.3 生产阶段应增加的能力

- `session_id` 持久化
- revision history 持久化
- learning event 日志表
- 人工审核后台
- 规则更新流程

## 十二、数据飞轮设计

### 12.1 必须记录的数据

1. 原始邮件文本
2. 初版 parse 结果
3. 每轮 user_feedback
4. 每轮 revise 后结果
5. 最终 confirm 结果

### 12.2 飞轮使用方式

#### 第一阶段：仅记录
- 保存全部修订轨迹
- 统计高频错误类型

#### 第二阶段：生成候选优化建议
- alias 候选
- 拆分规则候选
- 服务类型映射候选

#### 第三阶段：人工审核后吸收
- 更新 R2 alias
- 更新规则库
- 更新 few-shot 样例
- 更新 Prompt

### 12.3 不建议当前阶段做的事

不建议直接把用户单次修正自动写入正式规则库，避免污染正式知识。

## 十三、版本演进规划

### V1（当前开发版）
- parse / revise / confirm 三阶段完成
- 本地可运行
- JSON 输出稳定
- 规则优先
- next_questions 支持

### V2（生产初期版）
- 接入正式 R2 枚举
- 会话持久化
- 修订历史持久化
- 学习事件日志
- 人工确认界面

### V3（生产增强版）
- 增加字段级修订能力
- 支持更细粒度问答
- 增加更稳健的多服务项拆分
- 支持附件元信息联动

### V4（飞轮优化版）
- 离线统计高频修订
- 人工审核后批量更新 alias/规则
- few-shot 样例集优化
- 低置信场景接入 LLM fallback

### V5（高级版）
- 附件解析
- OCR / PDF / Word 支持
- 更强的多语言理解
- 自动推荐补问问题
- 精细化修订分类器

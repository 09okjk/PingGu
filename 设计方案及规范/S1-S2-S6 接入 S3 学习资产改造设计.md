# S1 / S2 / S6 接入 S3 学习资产改造设计

> 创建日期：2026-03-25  
> 最后修改：2026-03-25  
> 状态：分阶段实施中（Phase 1 计划确认）  
> 适用范围：S1 SearchHistoryCasesSkill / S2 AssessmentReasoningSkill / S6 GenerateReportSkill

---

## 一、文档目标

本文档描述如何将 **S3 LearningFlywheelSkill** 产出的学习资产，逐步接入到现有的：

- `S1 SearchHistoryCasesSkill`
- `S2 AssessmentReasoningSkill`
- `S6 GenerateReportSkill`

中，使系统从“只参考历史案例和规则”升级为：

> **同时参考历史案例 + 人工修订经验 + 已审核规则 + 已审核输出偏好**

同时，本文档不仅描述最终目标方案，也明确当前阶段的接入程度与后续演进计划，方便持续迭代。

---

## 二、当前定位说明

### 当前目标不是“一步到位全面接入”

本轮改造目标是：

> **先完成 S1 / S2 / S6 对 S3 的初步接入（Phase 1）**

即：

- S1：先能返回 `learning_samples`
- S2：先能接收 `learning_samples` 并生成 `learning_signals`
- S6：先能接收 `learning_samples` 并生成 `learning_summary`

### 当前阶段不做的内容

当前阶段明确**不做**：

- 不让 `learning_samples` 直接覆盖风险/工时/人数结果
- 不让未审核规则直接进入 S2 强逻辑
- 不让未审核偏好直接影响 S6 最终结构裁决
- 不做学习资产的复杂融合排序
- 不做批量聚类与高频模式自动升级
- 不做完整审核工作台

因此，本次接入是：

> **“弱耦合、低风险、可解释”的初步接入版**

---

## 三、总体改造原则

### 3.1 原则一：S3 资产是“增强输入”，不是“绝对事实”

S3 提供的是经验增强，不应直接覆盖：

- S5 已确认需求
- S2 已有稳定规则
- S1 结构化精确匹配结果

因此接入时应遵循：

```text
当前已确认需求
  >
正式已审核规则
  >
高相似历史案例
  >
高质量学习资产
  >
候选资产 / 弱信号
```

---

### 3.2 原则二：候选资产与已生效资产严格分层

#### 允许直接接入主流程的资产
- 高质量 `learning_samples`（作为补充参考）
- 已审核规则（后续接入）
- 已审核输出偏好（后续接入）

#### 当前阶段允许接入的 S3 资产
- `learning_samples`：仅作为补充参考，不强裁决

#### 当前阶段不直接接入的资产
- 未审核规则候选
- 未审核偏好候选
- 单次偶发修订模式

---

### 3.3 原则三：先做“参考增强”，再做“强逻辑融合”

建议分阶段实施：

#### Phase 1：初步接入（本轮目标）
- S1 增加修订经验样本检索/返回
- S2 增加 `learning_samples` 输入与 `learning_signals`
- S6 增加 `learning_samples` 输入与 `learning_summary`

#### Phase 2：正式生效资产接入
- S2 接入 approved rules
- S6 接入 approved report preferences

#### Phase 3：深度融合
- learning asset 排序融合
- 高风险场景自动增强审核提示
- learning 信号参与 confidence 计算
- 更细粒度策略调整

## 六、S1 改造设计：接入修订经验样本检索

---

### 6.1 改造目标

S1 当前只检索历史案例。Phase 1 改造后，希望它还能返回：

> “类似需求在过去被人工如何修订过”

---

### 6.2 当前阶段（Phase 1）接入范围

#### 本轮要做
- 在 S1 输出中增加 `learning_samples`
- learning_samples 作为单独补充字段返回
- 不影响原有 `results` 的返回结构与排序逻辑

#### 本轮不做
- 不把 learning_samples 混入历史案例主排序
- 不让 learning_samples 替代 `results`
- 不做复杂场景打分融合

---

### 6.3 输入复用

S1 无需增加新的对外必填参数，继续使用原有输入：

- `business_type`
- `service_desc_code`
- `service_type_code`
- `equipment_model_code`
- `task_description`
- `remark`
- `top_k`

---

### 6.4 Phase 1 检索逻辑建议

#### Step A：保持原有历史案例检索不变
原有：

```text
business_type + service_desc_code + service_type_code(+ relax)
```

#### Step B：新增 learning_samples 查询
新增从 `learning_samples` 中读取高质量样本：

筛选建议：
- `quality_score >= 0.75`
- `status in ('candidate', 'approved')`

匹配条件建议：
- `scenario.business_type`
- `scenario.service_desc_code`
- `scenario.service_type_code`

#### Step C：作为补充字段返回
结果放在：

```json
"learning_samples": [...]
```

不参与当前 `results` 主排序。

---

### 6.5 S1 输出结构（Phase 1）

当前建议输出：

```json
{
  "results": [...],
  "learning_samples": [
    {
      "sample_id": "LF-AT202603250001",
      "scenario_match_reason": "命中：业务归口(轮机) + 服务描述 + 服务类型",
      "quality_score": 0.92,
      "revision_summary": "风险等级上调，并补充备件等待影响",
      "source_task_id": "AT202603250001",
      "status": "candidate"
    }
  ]
}
```

---

### 6.6 Phase 2 / 3 预留

#### Phase 2
- 增加 `approved` 样本优先
- 增加更多匹配字段

#### Phase 3
- 与 `results` 做统一排序融合
- 支持 learning sample 摘要压缩与排序分层

## 七、S2 改造设计：接入 learning_samples 并输出 learning_signals

---

### 7.1 改造目标

S2 当前主要基于：

- requirement
- history_cases
- R2 / R3 / R5 / R6

Phase 1 改造后，希望它还能参考：

- `learning_samples`

但只做**提示增强**，不做强逻辑覆盖。

---

### 7.2 当前阶段（Phase 1）接入范围

#### 本轮要做
- S2 输入增加 `learning_samples`
- 从 learning_samples 中提取弱信号
- 输出：
  - `learning_signals`
  - `learning_references`
- 在 `warnings` / `reasoning_trace` 中体现学习提示

#### 本轮不做
- 不直接修改 risk_results / workhour_results / manpower_result 的值
- 不接 approved rules
- 不根据 learning_samples 自动改变总人数/总工时

---

### 7.3 输入结构（Phase 1）

建议升级为：

```json
{
  "requirement": {},
  "history_cases": [],
  "learning_samples": [],
  "options": {}
}
```

说明：
- `learning_samples` 由 S1 返回后透传
- 当前阶段是可选字段，缺失也应正常运行

---

### 7.4 当前阶段内部逻辑

在 Step 1 上下文准备后新增：

```text
Step 1.1：吸收 learning_samples 摘要
Step 1.2：提取 learning_signals
```

#### 示例 signal
- `similar_revisions_often_raise_risk_level`
- `similar_revisions_often_add_spare_part_notes`
- `similar_revisions_often_increase_hours`

---

### 7.5 当前阶段作用方式

#### 风险推理
- 增加 warning
- 增加 reasoning_trace
- 可降低 confidence
- 不直接改 risk level

#### 工时推理
- 增加 warning / note
- 可降低 confidence
- 不直接改 suggested_hours

#### 人数推理
- 增加 warning / reasoning_trace
- 不直接改 total_persons

---

### 7.6 S2 输出结构（Phase 1）

新增：

```json
{
  "learning_signals": [
    "similar_revisions_often_raise_risk_level"
  ],
  "learning_references": [
    "LF-AT202603250001"
  ]
}
```

并在：

- `reasoning_trace`
- `warnings`

中体现。

---

### 7.7 Phase 2 / 3 预留

#### Phase 2
- 接入 approved rules
- learning_signals 与 approved rules 共同参与 reasoning basis

#### Phase 3
- learning 信号参与 confidence 计算
- 在特定场景下适度参与推理参数调整

## 八、S6 改造设计：接入 learning_samples 并输出 learning_summary

---

### 8.1 改造目标

S6 当前负责把：

- requirement
- history_cases
- assessment_result

组织成最终报告草稿。

Phase 1 改造后，希望它还能：

- 使用 `learning_samples` 生成学习摘要
- 增强 `review_focus`
- 增强 `warnings`

但不直接重构报告主结构。

---

### 8.2 当前阶段（Phase 1）接入范围

#### 本轮要做
- S6 输入增加 `learning_samples`
- 输出增加 `learning_summary`
- 基于 learning_samples 增强 `review_focus`

#### 本轮不做
- 不接 approved preferences
- 不动态重排整份报告结构
- 不自动新增任务条目
- 不直接填充事实字段

---

### 8.3 输入结构（Phase 1）

建议升级为：

```json
{
  "requirement": {},
  "history_cases": [],
  "learning_samples": [],
  "assessment_result": {},
  "options": {}
}
```

---

### 8.4 当前阶段内部逻辑

新增轻量步骤：

```text
Step 1.1：读取 learning_samples
Step 1.2：提取常见 revision_summary 关键词
Step 1.3：生成 learning_summary
Step 1.4：增强 review_focus
```

---

### 8.5 当前阶段增强点

#### review_focus
例如：
- “类似场景中人工常补充备件等待影响，请重点确认。”
- “类似场景中人工常上调风险等级，请重点确认风险判断是否偏低。”

#### warnings
当 learning_samples 与当前草稿差距较大时，可增加弱 warning。

---

### 8.6 S6 输出结构（Phase 1）

新增：

```json
{
  "learning_summary": {
    "used_learning_sample_ids": [
      "LF-AT202603250001"
    ],
    "applied_learning_hints": [
      "similar_cases_often_add_spare_part_confirmation",
      "similar_cases_often_raise_risk_level"
    ]
  }
}
```

---

### 8.7 Phase 2 / 3 预留

#### Phase 2
- 接入 approved report preferences
- 影响章节顺序与重点呈现方式

#### Phase 3
- 动态调整 summary / report_table 的组织结构
- 更细粒度地影响字段展示策略

## 十二、风险与控制措施

### 风险 1：learning_samples 噪声过大
**控制措施**：
- 仅取 `quality_score >= 0.75`
- 当前阶段只做补充参考

### 风险 2：S2 被学习样本误导
**控制措施**：
- 当前阶段只输出 signal / warning
- 不直接改 risk/hour/manpower 数值

### 风险 3：S6 被过度偏好化
**控制措施**：
- 当前阶段只增强 `review_focus`
- 不重排核心报告结构

## 一、背景与目标

### 背景

服务贸易人员根据客户的原始需求制作**服务需求单**，评估人员需要根据需求单内容人工完成**服务评估报告**。现有流程依赖评估人员的个人经验，效率较低，且经验难以沉淀传承。

### 目标

构建一个**智能评估 Agent**，作为上层 Agent 的编排模块（参考 OpenClaw Skill 模式），通过多个细粒度 Skill 协作，实现：

- 自动读取并理解服务需求单（含非结构化备注/原始邮件）
- 渐进式检索历史相似评估案例，辅助生成评估报告草稿
- 对每个生成字段标注置信度，引导评估人员高效审核
- 随使用积累持续提升推荐精准度（数据飞轮）

### 当前数据基础

|------|------|------|
| 历史评估数据 | 8000 条，已清洗入库到 PostgreSQL | ✅ 完成 |

| 工种配置 | 25 工种（JN0001-JN0031） | ✅ 完成 |

| 员工能力数据 | 暂无，评估人员凭经验判断，二期接入 | 📅 计划 |

## 二、业务流程概述

```text
客户原始需求（邮件等）
        ↓
服务贸易人员制作【服务需求单】
        ↓
评估人员提交需求单至 Agent
        ↓
智能评估 Agent 编排处理
├── ParseRequirementSkill：解析结构化字段 + NLU 备注
├── SearchHistoryCasesSkill：渐进式检索历史相似案例
├── AssessmentReasoningSkill：统一完成风险识别、工时估算、人力推理
├── GenerateReportSkill：整合结果，生成评估报告草稿（含置信度标注）
├── DialogIntentDetector：识别审核对话意图，管理状态机，触发 S3 学习
└── LearningFlywheelSkill：采集人工修订、提炼经验资产、反哺后续评估
└── ReviewPersistenceSkill：评估单状态持久化与恢复（Redis）
        ↓
评估人员审核 & 修改（多轮对话）
        ↓
最终评估报告确认输出
        ↓
差异回流 → 学习资产沉淀 → 历史数据库/规则候选更新（飞轮）
```

## 四、服务评估报告字段定义（输出）

### 4.1 风险提示

- 从已有风险库中选择匹配项
- 若风险库中无合适条目，评估人员可手动输入新增
- 支持多条

### 4.2 施工任务

- 支持多条
- 每条内容：具体的工作内容描述，用于指导施工人员

### 4.3 施工人数及工时

支持多条，每条包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| 详细工作内容 | 文本 | 具体工作细项描述 |
| 单位 | 枚举（选填） | 个、台、把等 |
| 数量 | 数字 | 面向的设备数量 |
| 工种 | 枚举（有穷） | 如：电气工程师、轮机工程师等 |
| 职级 | 枚举（按工种细分，有穷） | 不同工种有不同职级体系 |
| 人数 | 数字 | 该项工作所需人数 |
| 单人工时 | 数字（小时） | **注意**：指整体施工从开始到结束的时长，非个人完成该任务的时间。即使该任务提前完成，只要整体施工未结束，工时持续计算。 |

### 4.4 综述

| 字段 | 类型 | 说明 |
|------|------|------|
| 是否航修 | 布尔 | 是否属于航修类型 |
| 总小时数 | 数字 | 施工开始到结束的总时长，**非所有人工时相加** |
| 总人数 | 数字 | 最少所需总人数，**非各任务人数直接相加**（见第八节专项说明） |

### 4.5 需要工具

支持多条，每条包含：

| 字段 | 类型 |
|------|------|
| 工具名称 | 文本 |
| 工具类型 | 枚举（有穷） |
| 计量单位 | 枚举 |
| 需要数量 | 数字 |

### 4.6 耗材

支持多条，每条包含：

| 字段 | 类型 |
|------|------|
| 耗材名称 | 文本 |
| 耗材型号 | 文本（必填） |
| 计量单位 | 枚举 |
| 数量 | 数字 |

### 4.7 专用工具

支持多条，每条包含：

| 字段 | 类型 |
|------|------|
| 专用工具名称 | 文本 |
| 专用工具型号 | 文本 |
| 计量单位 | 枚举 |
| 数量 | 数字 |

### 4.8 设备/备件需求

- 描述本次服务需要提供的设备或备件
- 注意：若需求方自身有多余设备或备件，则无需填写（我方不提供）

## 五、整体架构设计

### 架构定位

这是一个**知识增强型结构化推理 Agent**，由上层编排层调度多个细粒度 Skill 协作完成评估任务。核心能力层次如下：

```text
┌─────────────────────────────────────────────────────────────┐
│                     智能评估 Agent（编排层）                  │
│                                                             │
│   ┌─────────────┐    ┌──────────────┐    ┌──────────────┐   │
│   │  理解与提取  │ →  │  检索与匹配  │ →  │   推理与生成  │   │
│   │  (NLU 层)   │    │ (Search 层)  │    │ (Reasoning / │   │
│   │             │    │              │    │ Generation)  │   │
│   └─────────────┘    └──────────────┘    └──────────────┘   │
│                              ↓                              │
│                     学习与反哺（Flywheel）                   │
└─────────────────────────────────────────────────────────────┘
         ↑ 输入：服务需求单                ↓ 输出：评估报告草稿
```

### Agent 在体系中的位置

```text
上层 Agent（对话/工作流编排）
        ↓ 调用
┌────────────────────────────────────────────────────────┐
│              智能评估 Agent（编排层）                    │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │               Skills（主动执行）                   │  │
│  │  ParseRequirementSkill                           │  │
│  │  SearchHistoryCasesSkill                         │  │
│  │  AssessmentReasoningSkill                        │  │
│  │  GenerateReportSkill                             │  │
│  │  LearningFlywheelSkill                           │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │             References（静态/规则注入）            │  │
│  │  R2 枚举字典与业务规则库                         │  │
│  │  R3 风险规则库                                   │  │
│  │  R4 工具/耗材推荐模板（第二期）                   │  │
│  │  R5 工时经验规则库                               │  │
│  │  R6 人力配置规则库                               │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
        ↓ 返回
结构化评估报告（含置信度标注）
```

### Skills vs References 划分原则

| 类型 | 适用场景 | 特征 |
|------|---------|------|
| **Skill** | 需要统一编排、多步推理、内部逻辑执行 | 主动执行，有流程控制 |
| **Reference** | 静态规则、经验数据、枚举校验、模板套用 | 被动注入 context / 查询，不主动执行 |

### 本次调整后的划分说明

原方案中的：

- `S2 MatchRisksSkill`
- `S3 EstimateWorkHoursSkill`
- `S4 EstimateManpowerSkill`

现整合为统一的 **`S2 AssessmentReasoningSkill`**。

同时，为了把“人工修订经验 → 可复用知识资产 → 下一轮优化”形成独立、可控、可回滚的闭环，新增：

- **`S3 LearningFlywheelSkill`**：负责采集修订差异、归因打标、提炼学习样本/规则候选/输出偏好，并将结果沉淀到学习资产库中。

整合后的设计原则为：

- **AssessmentReasoningSkill** 负责统一执行风险识别、工时估算、人力推理
- **LearningFlywheelSkill** 负责评估完成后的经验沉淀与后续反哺
- 原三者所依赖的专项知识下沉为三个 Reference：
  - `R3 风险规则库`
  - `R5 工时经验规则库`
  - `R6 人力配置规则库`

这样既减少主链路 Skill 数量和编排复杂度，也把“当前结论生成”和“未来能力增强”清晰分离。

### S2：AssessmentReasoningSkill ⭐ 最高优先级

**定位**：统一执行风险识别、工时估算与人力配置推理的评估推理 Skill。  
**整合来源**：原 `S2 MatchRisksSkill` + 原 `S3 EstimateWorkHoursSkill` + 原 `S4 EstimateManpowerSkill`。  
**实施优先级**：MVP 必须。  
**当前版本**：v1.1.0（2026-03-24 更新）

#### 整合原因

原方案中的风险匹配、工时估算与人数推理，本质上都属于同一层"评估推理能力"，且存在明显依赖关系：

```text
结构化需求 + 历史案例
    ↓
风险识别
    ↓
工时估算
    ↓
人数推理
    ↓
交给 S6 组织最终报告
```

因此在当前阶段，采用"**1 个 Skill + 3 个专项 Reference**"的方式更合理：

- 统一输入输出
- 降低编排复杂度
- 提高 MVP 落地效率
- 保留未来按复杂度再拆分的空间

#### 核心职责

输入：
- S5 输出的已确认结构化需求单
- S1 输出的历史相似案例
- R2 / R3 / R5 / R6 提供的规则与经验数据

输出：
- 风险推理结果
- 工时推理结果
- 人数推理结果
- 各域置信度
- 推理说明

#### 职责边界

AssessmentReasoningSkill：

- **负责输出推理结果**
- **不负责最终施工任务组织**
- **不负责最终报告字段组装**

最终任务组织与报告输出由 `S6 GenerateReportSkill` 负责。

#### 当前阶段实现策略

##### 风险推理
- 结构化条件匹配
- remark 关键词触发
- 历史案例风险补强
- 输出风险提示 + 建议措施 + 置信度
- **数据库规则**：`pinggu.risk_rules` (22 条)

##### 工时推理
- 当前历史统计数据不足
- 先采用"**单值工时 + 置信度**"的简化方案
- 初始工时经验库完全来自历史数据统计
- 后续再升级为区间估算（如 P50/P80/P95）
- **数据库规则**：`pinggu.workhour_rules` (4 条，需扩充)

##### 人数推理
- 按同工种内高职级覆盖低职级规则执行
- 跨工种不可替代
- 当前先采用"**可串行则复用**"的简化策略
- 输出理论最小总人数
- 当前阶段**不输出保守方案**
- **数据库规则**：`pinggu.manpower_level_cover_rules` (1146 条，自动生成)

- 风险匹配需要独立动态库和复杂排序
- 工时估算升级为复杂统计模型
- 人数推理接入真实员工能力和时间窗口约束

### S3：LearningFlywheelSkill ⭐ 中高优先级

**定位**：评估完成后的学习飞轮 Skill，负责采集“初稿 → 人工修订 → 最终确认”的全过程差异，提炼学习样本、规则候选和输出偏好，并反哺后续评估流程。  
**实施优先级**：第二期建议优先落地。  
**触发方式**：后台自动触发，不阻塞主评估链路。  
**当前状态**：📅 设计新增（2026-03-25）

#### 设计动机

当前 S5/S1/S2/S6 已可以支撑“生成评估草稿”，但系统要持续变得更像有经验的工务人员，必须把人工修订过程中的高价值信号结构化沉淀下来。

因此新增 `S3 LearningFlywheelSkill`，核心不是替代评估，而是：

1. **采集**：记录初稿、终稿、对话修订、字段改动与上下文版本信息
2. **提炼**：把一次修订抽象为可复用的学习资产
3. **反哺**：在下一次评估中为 S1/S2/S6 提供额外参考上下文

#### 核心职责

输入：
- 原始需求与已确认需求项（来自 S5）
- 历史检索结果（来自 S1）
- 评估推理结果（来自 S2）
- 初始评估报告草稿与最终确认报告（来自 S6 / 人工修订）
- 人机对话修订记录、字段编辑动作、Skill/Prompt/Reference 版本信息

输出：
- revision diff（结构化差异）
- feedback tags（修订归因标签）
- learning sample（高质量学习样本）
- rule candidates（候选规则）
- report preference candidates（输出偏好候选）
- next_step_actions（后续存储/审核动作建议）

#### 职责边界

LearningFlywheelSkill：

- **不直接修改当前评估结论**
- **不在当前链路中重新计算风险/工时/人数**
- **不自动将单次修订直接写入正式规则库**
- **只负责未来优化所需的记录、归因、提炼和反哺**

这意味着：

- 当前任务的终稿由人工确认负责
- 未来任务的优化由 S3 的沉淀资产逐步提供支撑

#### 学习资产分层

S3 当前建议沉淀三类资产：

1. **修订经验样本**
   - 用于后续相似任务检索
   - 优先支持 S1 的“修订经验参考”能力

2. **候选规则**
   - 用于高频、稳定修订模式的规则化沉淀
   - 需管理员/专家审核通过后方可进入正式规则体系

3. **输出偏好**
   - 用于学习不同业务场景下常见的报告结构、必备字段、表达偏好
   - 优先反哺 S6 的报告组织与审核引导

#### 对其他 Skill 的反哺方式

- **反哺 S1**：新增“相似人工修订案例”检索参考
- **反哺 S2**：读取已审核规则、修订高频提示、经验摘要
- **反哺 S6**：读取场景化输出偏好、常见补充项、审核重点模板

#### 实现原则

- 后台自动运行，对一线评估人员尽量无感
- 候选资产与已生效资产必须分层管理
- 优先学习共性修订模式，不放大偶发个体偏好
- 所有学习结果必须可追溯、可审核、可回滚

> 详细设计见《S3 - LearningFlywheelSkill 详细设计》。

### S4：DialogIntentDetector ⭐ 高优先级

**定位**：工务审核对话的意图识别与状态机管理 Skill，负责识别修改/确认/取消意图，管理审核状态流转，并在二次确认后构造 S3 学习输入。  
**实施优先级**：MVP 增强（v1.0.0 基础版，v1.1.0 状态持久化版）。  
**当前版本**：v1.1.0（2026-03-26 状态持久化版）

#### 核心职责

输入：
- 工务对话消息
- 任务 ID / 组织 ID / 用户 ID
- 当前状态（可选）

输出：
- 意图识别结果（MODIFY / READY_TO_CONFIRM / CONFIRM / CANCEL）
- 状态流转建议
- S3 输入构造（CONFIRM 时）
- 修订历史记录

#### 4 种意图类型

| 意图 | 触发词示例 | 状态转换 | 后续动作 |
|------|-----------|---------|---------|
| `MODIFY` | "修改/调整/改成/风险调高" | REVIEW_IN_PROGRESS → REVIEW_IN_PROGRESS | 调用 S6 修改报告 |
| `READY_TO_CONFIRM` | "好了/可以了/没问题" | REVIEW_IN_PROGRESS → CONFIRMATION_PENDING | 展示修订摘要 |
| `CONFIRM` | "确认/通过/同意/交付" | CONFIRMATION_PENDING → COMPLETED | 调用 S3 学习飞轮 |
| `CANCEL` | "取消/放弃/算了" | CONFIRMATION_PENDING → REVIEW_IN_PROGRESS | 返回审核中状态 |

#### 3 状态状态机

```
REVIEW_IN_PROGRESS（审核中）
    ↓ (READY_TO_CONFIRM)
CONFIRMATION_PENDING（待确认）
    ↓ (CONFIRM)
COMPLETED（完成）→ 调用 S3
```

#### Redis 持久化（v1.1.0+）

- 状态保存到 Redis（TTL 1 小时）
- 进程重启后自动恢复
- 未完成列表查询
- 用户首次对话主动提示

> 详细设计见《S4 - DialogIntentDetector 详细设计》。

### S7：ReviewPersistenceSkill ⭐ 中高优先级

**定位**：评估单状态持久化与恢复管理 Skill，支持进程重启后无缝恢复审核流程，提供分布式锁防止并发修改。

#### 核心职责（五阶段）

**Phase 1-2: 基础持久化**
- 状态持久化到 Redis
- 状态恢复
- 未完成列表查询

**Phase 3: 深度集成**
- 与 ReviewStateMachine 深度集成
- 自动同步状态机状态

**Phase 4: 自动恢复**
- Agent 启动时自动扫描未完成状态
- 重建状态机实例

**Phase 5: 主动通知**
- 用户首次对话时主动提示未完成任务
- 支持用户选择继续或开始新任务

**优化特性**
- 分布式锁防止并发修改

#### Redis 数据结构

```
review_state:{task_id}          # 单个评估单状态（Hash，TTL 1 小时）
user_pending_tasks:{user_id}    # 用户未完成列表（Set，TTL 24 小时）
global_pending_tasks            # 全局未完成列表（ZSet，TTL 24 小时）
lock:{resource_name}            # 分布式锁（String，TTL 30 秒）
```

- **S7 是 S4 的持久化层**
- **S4 每次交互后自动调用 S7 保存状态**
- **S4 重启后从 S7 恢复状态**

## 八、总人数计算专项设计

> 总人数推理现作为 `S2 AssessmentReasoningSkill` 的内部推理域之一执行，其规则依赖 `R6 人力配置规则库`。

### 核心规则

- 总人数 ≠ 各任务人数直接相加
- 高职级人员完成串行任务后，可接续承担低职级任务，从而节省人力
- 跨工种不可替代

### 当前阶段简化策略

当前阶段先采用以下策略：

1. **同工种内高职级覆盖低职级**
2. **跨工种不可替代**
3. **若任务可串行，则允许人员复用**
4. 输出**理论最小总人数**
5. **当前阶段不输出保守方案**
6. 后续保留升级到“显式时间窗口建模”的空间

### 计算模型

```text
输入：
  - 任务集合（由 S1 历史案例参考 + S2 工时推理结果提供依据）
  - 职级能力覆盖表（来自 R6）
  - 通用业务规则（来自 R2）

推理步骤：
  1. 识别任务所属工种与最低职级要求
  2. 判断是否可串行复用
  3. 在同工种高职级覆盖低职级约束下求最小人员配置
  4. 输出理论最小总人数及说明

输出：
  - 最小总人数方案
  - 推理过程说明（供评估人员审核）
```

### 示例说明

```text
场景：
  - 任务1：高职级电气工程师，2人，持续2小时，仅高职级可做
  - 任务2：电气工程师（高/低职级均可），持续6小时，需3人
  - 两任务可串行执行

推理过程：
  - 任务1结束后，2名高职级工程师可继续承担任务2
  - 任务2共需3人，因此只需再补充1名低职级人员
  - 最小总人数 = 2（高职级） + 1（低职级） = 3人
```

### 当前阶段边界

由于尚未接入真实员工能力数据（S7 未建设），当前人数结果只代表：

> **在既定规则下的理论最优配置建议**

实际排班仍需评估人员结合现场人员情况进行确认。

## 九、多语言处理策略

**支持语言**：英文（主）、简体中文、繁体中文、韩语、日语

### 核心方案

使用**多语言 Embedding 模型**（推荐：`multilingual-e5-large` 或 OpenAI `text-embedding-3-large`），一个模型覆盖全部语言，无需分语言维护多套索引。

### 处理流程

```text
输入层：保留原文，不做翻译
    ↓
Embedding 层：多语言模型统一向量化
    ↓
检索层：跨语言语义匹配
（例：日语输入 → 可匹配英文历史记录中的相似案例）
    ↓
生成层：输出语言跟随输入语言
（用户用什么语言提交需求单，AI 用什么语言回应）
```

### 业务价值

不同语言的客户需求单，可以无缝匹配到任意语言的历史案例，最大化历史数据的利用价值。

### Agent 自动编排 `assess()` 接口

**核心原则**：`assess()` 不是独立开发的接口，而是由上层 Agent 根据各 Skill 的输入输出契约自动编排生成。

```python
# 上层 Agent 自动编排示例
async def assess(requirement_input: Dict) -> Dict:
    """
    智能评估主接口 - 由 Agent 自动编排
    
    编排逻辑：
    1. 调用 S5 解析需求单
    2. 对每个服务项调用 S1 检索历史案例
    3. 调用 S2 进行评估推理
    4. 调用 S6 生成评估报告
    5. 初始化 S4 状态机（等待用户对话）
    6. 调用 S7 保存初始状态
    """
    
    # Step 1: S5 需求解析
    s5_result = await call_skill("parse-requirement-skill", {
        "action": "parse",
        "input": requirement_input["email_text"],
        "refs": "references/r2-enums.json"
    })
    
    requirements = s5_result["data"]["service_items"]
    
    # Step 2-4: 对每个服务项并行处理
    reports = []
    for req in requirements:
        # S1 历史检索
        s1_result = await call_skill("search-history-cases-skill", {
            "requirement": req
        })
        
        # S2 评估推理
        s2_result = await call_skill("assessment-reasoning-skill", {
            "requirement": req,
            "history_cases": s1_result["data"]["cases"]
        })
        
        # S6 报告生成
        s6_result = await call_skill("generate-report-skill", {
            "requirement": req,
            "history_cases": s1_result["data"]["cases"],
            "assessment_result": s2_result["data"]
        })
        
        reports.append(s6_result["data"])
    
    # Step 5: 初始化 S4 状态机
    task_id = generate_task_id()
    await call_skill("dialog-intent-detector", {
        "action": "init_state",
        "task_id": task_id,
        "report": reports
    })
    
    # Step 6: S7 状态持久化
    await call_skill("review-persistence-skill", {
        "action": "save",
        "task_id": task_id,
        "state": "REVIEW_IN_PROGRESS"
    })
    
    return {
        "success": True,
        "data": {
            "task_id": task_id,
            "reports": reports,
            "next_action": "wait_for_user_review"
        },
        "error": None
    }
```

### 输出格式设计原则

- ✅ 每个字段附带 `confidence` 置信度（`high` / `medium` / `low`）
- ✅ 每个字段附带 `source`（来源历史记录 ID 或规则说明）
- ✅ 结构化 JSON 输出，便于上层 Agent 进一步处理或前端渲染
- ✅ 学习类输出必须区分"候选资产"与"已生效资产"
- ✅ **所有 Skill 输出包含 `success / data / error` 标准字段**

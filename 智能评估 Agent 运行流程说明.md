# 智能评估 Agent 运行流程说明

> 文档版本：1.0  
> 最后更新：2026-03-25  
> 适用对象：业务人员、管理人员、最终用户

---

## 一、系统概述

智能评估 Agent 是一套自动化系统，用于将**船舶服务需求邮件**自动转换为**结构化评估报告**。

### 核心价值

| 传统方式 | 智能评估 Agent |
|---------|---------------|
| 人工阅读邮件，耗时 30-60 分钟 | 自动解析，30 秒完成 |
| 凭经验估算工时和人数 | 参考历史案例 + 规则推理 |
| 报告格式不统一 | 标准化输出模板 |
| 经验难以沉淀 | 自动学习人工修订经验 |

### 系统组成

```
┌───────────────────────────────────────────────────────────┐
│                    智能评估 Agent 系统                     │
│                                                           │
│   S5        S1         S2         S6         S3           │
│   需求解析 → 历史检索 → 评估推理 → 报告生成 → 学习飞轮       │
│                                                           │
│   (5 个核心模块协同工作)                                    │
└───────────────────────────────────────────────────────────┘
```

## 二、完整运行流程

### 阶段一：提交需求

**用户操作**：发送服务需求邮件

**示例邮件内容**：
```
主题：主机检测服务需求

设备名称：主机
设备型号：MAN B&W-9S90ME-C9.2-TII
服务类型：检测
备注：内含船厂常规工作，可能存在交叉作业与备件等待
```

---

### 阶段二：需求解析（S5）

**系统自动执行**

系统读取邮件后，自动识别关键信息：

| 识别项 | 提取结果 |
|-------|---------|
| 业务类型 | 轮机 |
| 服务描述 | 二冲程柴油机 |
| 服务类型 | 检测 |
| 设备名称 | 主机 |
| 设备型号 | MAN B&W-9S90ME-C9.2-TII |

**如有歧义，系统会提问**：
> "检测到可能同时包含'检测'和'维修'服务，请确认具体需要哪种服务？"

**产出**：结构化需求单

---

### 阶段三：历史检索（S1）

**系统自动执行**

系统在历史数据库中查找相似案例：

```
正在搜索历史案例...
找到 5 个相似案例：

1. RH-2025-0009611001 - 主机常规坞修保养
   工时：96 小时 | 人数：9 人 | 工期：12 天

2. RH-2024-0008452003 - 主机振动检测
   工时：80 小时 | 人数：6 人 | 工期：10 天

3. ...
```

**匹配依据**：
- 业务类型相同（轮机）
- 服务描述相似（柴油机相关）
- 设备型号匹配（MAN B&W）
- 任务描述相似度

**产出**：历史案例参考列表

---

### 阶段四：评估推理（S2）

**系统自动执行**

系统综合需求和历史案例，进行三项推理：

#### 1. 风险识别
```
⚠️ 识别到以下风险：

中风险：船厂交叉作业导致工期延误
  依据：备注中提到"交叉作业"
  
低风险：备件等待可能影响进度
  依据：备注中提到"备件等待"
```

#### 2. 工时估算
```
建议工时：96 小时
  - 历史参考：类似案例平均 90-100 小时
  - 风险修正：+6 小时（交叉作业影响）
```

#### 3. 人力配置
```
建议人数：3 人
  - 工种配置：技工 2 人 + 助理 1 人
  - 依据：可串行作业，复用历史配置
```

**产出**：评估推理结果（风险 + 工时 + 人力）

---

### 阶段五：报告生成（S6）

**系统自动执行**

系统组装完整评估报告：

```
┌─────────────────────────────────────────┐
│         评估报告（初稿）                  │
├─────────────────────────────────────────┤
│ 【风险提示】                             │
│ 1. 船厂交叉作业可能导致工期延误 [中风险]  │
│                                          │
│ 【施工任务】                             │
│ 主机检测服务，型号 MAN B&W-9S90ME-C9.2   │
│                                          │
│ 【总计】                                 │
│ 总工时：96 小时                          │
│ 总人数：3 人                             │
│                                          │
│ 【审核关注点】                           │
│ - 请确认交叉作业时间协调                  │
│ - 请确认备件到位情况                      │
└─────────────────────────────────────────┘
```

**产出**：评估报告初稿（状态：待审核）

---

### 阶段六：人工审核修订

**用户操作**：审核并修改报告

这是**关键环节**，用户可以根据实际经验调整报告：

#### 典型修改场景

| 修改项 | 初稿值 | 用户修改为 | 修改原因 |
|-------|-------|-----------|---------|
| 风险等级 | 中风险 | **高风险** | 现场交叉作业复杂 |
| 总工时 | 96 小时 | **110 小时** | 增加备件等待时间 |
| 总人数 | 3 人 | **4 人** | 需要额外协助 |
| 审核关注点 | 2 条 | **增加 1 条** | 补充备件提供方确认 |

#### 修改方式

1. **直接编辑** - 点击字段修改数值
2. **下拉选择** - 选择风险等级等选项
3. **添加内容** - 补充审核关注点、备注等

#### 确认提交

修改完成后，点击 **"确认"** 按钮：

```
✓ 报告已确认
  状态：已确认 (confirmed)
  修改记录：已保存
```

**产出**：
- 最终版评估报告
- 修改记录（哪些字段被修改）
- 对话反馈（用户的修改说明）

---

### 阶段七：学习飞轮（S3）

**系统自动执行**（用户无感知）

系统在报告确认后，自动分析人工修订过程：

#### 1. 提取修订差异

```
检测到以下修改：
┌──────────────┬──────────┬──────────┐
│ 字段         │ 修改前   │ 修改后   │
├──────────────┼──────────┼──────────┤
│ 风险等级     │ medium   │ high     │
│ 总工时       │ 96       │ 110      │
│ 总人数       │ 3        │ 4        │
│ 审核关注点   │ 2 条     │ 3 条     │
└──────────────┴──────────┴──────────┘
```

#### 2. 归因分析

系统分析"为什么用户要这样修改"：

```
识别到修订原因：
✓ RISK_UNDER_ESTIMATED - 初始风险低估
✓ WORKHOUR_UNDER_ESTIMATED - 初始工时低估
✓ MANPOWER_UNDER_ESTIMATED - 初始人力低估
✓ MISSING_DIMENSION - 缺失考虑维度（备件）
```

#### 3. 质量评分

系统评估这次修订的学习价值：

```
学习样本质量评分：0.95 (高价值)

评分依据：
✓ 任务 ID 可追溯
✓ 初稿与终稿结构完整
✓ 存在明确修订差异
✓ 可识别归因标签
✓ 终稿状态明确
✓ 包含对话上下文
```

> **评分 ≥ 0.55** 的样本会被存储，用于未来优化

#### 4. 生成候选规则

系统从修订中提取可复用的规则：

```
生成 3 条候选规则：

规则 1：风险等级下限规则
  触发条件：业务类型=轮机 + 备注含"交叉作业"
  建议：风险等级不应低于 medium
  置信度：78%

规则 2：必填维度规则
  触发条件：业务类型=轮机 + 船型=散货船
  建议：报告必须包含"设备/备件需求"章节
  置信度：72%

规则 3：审核关注点增强规则
  触发条件：备注含"交叉作业"或"备件"
  建议：必须包含 suggested_action 和 review_focus
  置信度：70%
```

#### 5. 识别输出偏好

系统学习用户的报告偏好：

```
识别到 1 条偏好：

偏好：结构化审核提示
  场景：轮机业务
  内容：保留更多结构化审核关注点
  置信度：55%
```

#### 6. 持久化存储

学习成果被保存到两个地方：

| 存储位置 | 内容 | 用途 |
|---------|------|------|
| **本地文件** | 完整修订记录 JSON | 开发调试、审计追溯 |
| **PostgreSQL 数据库** | 结构化学习资产 | 统计分析、规则审核 |

**数据库表结构**：
```
learning_revision_records    - 修订记录 (1 条)
learning_feedback_tags       - 反馈标签 (6 条)
learning_samples            - 学习样本 (1 条)
learning_rule_candidates    - 候选规则 (3 条)
learning_report_preferences - 输出偏好 (1 条)
```

**产出**：
- 学习资产已入库
- 候选规则待审核
- 下次评估可参考

## 三、知识闭环（未来版本）

学习到的知识如何反哺系统？

```
┌─────────────────────────────────────────────────────────────┐
│                      知识反哺流程                            │
│                                                             │
│   管理员审核候选规则                                         │
│         ↓                                                   │
│   审核通过 → 生效为正式规则 → 更新 S2 推理规则                │
│         ↓                                                   │
│   聚合高频偏好 → 优化 S6 报告模板                            │
│         ↓                                                   │
│   分析常见低估场景 → 优化 S1 检索策略                        │
│                                                             │
│   形成闭环：S5→S1→S2→S6→人工修订→S3→知识反哺→S1/S2/S6 优化    │
└─────────────────────────────────────────────────────────────┘
```

### 实际效果示例

**场景**：多位审核员都上调了某类任务的风险等级

```
第 1 次：审核员 A 将"主机检测"风险从 medium 改为 high
第 5 次：审核员 B 同样修改
第 10 次：审核员 C 同样修改

↓

S3 识别到规律：
  "主机检测 + 交叉作业"场景下，80% 的案例需要上调风险

↓

管理员审核通过：
  新增规则：该场景下风险等级下限为 high

↓

系统优化：
  未来同类任务的初稿风险等级自动设为 high
  减少人工修改次数
```

## 四、关键问答

### Q1: 系统会完全替代人工吗？

**不会**。系统设计为"人机协同"：
- 系统生成初稿（节省时间）
- 人工审核修订（保证质量）
- 系统学习经验（持续优化）

### Q2: 人工修改会被"记住"吗？

**会**。S3 学习飞轮会：
- 记录每次修改
- 分析修改原因
- 提取可复用规则
- 优化未来初稿质量

### Q3: 如果修改错了怎么办？

**可追溯**：
- 所有修改记录都保存在数据库
- 可以查看"谁在什么时候改了什么"
- 支持审计和复盘

### Q4: 系统如何保证学习质量？

**质量过滤**：
- 只有评分 ≥ 0.55 的修订才会被存储
- 候选规则需要管理员审核才能生效
- 避免错误经验被放大

### Q5: 一次评估需要多长时间？

| 阶段 | 耗时 |
|------|------|
| S5-S6 自动生成 | 30-60 秒 |
| 人工审核修订 | 5-15 分钟（因复杂度而异） |
| S3 自动学习 | 5-10 秒（后台执行） |
| **总计** | **约 6-16 分钟** |

相比传统方式（30-60 分钟），效率提升约 **60-75%**
